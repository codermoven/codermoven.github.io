###Docker初体验

​ Docker 是一个开源的基于 LXC 的高级应用容器引擎，让开发者可以打包他们的应用以及依赖包到一个可移植的容器中，然后发布到任何流行的 Linux 机器上，也可以实现虚拟化。容器是完全使用沙箱机制，相互之间不会有任何接口。（不会总结，但是觉得百度说的蛮好）;

​ 综合以上概念，我们可以看出docker是一个应用容器的引擎，其中关键词有两个，应用容器，引擎，引擎自不必说，最为关键的是应用容器，也就是说docker为应用程序的运行提供独立的容器，而且该容器还可以移植，想想都觉的美爆了，应用程序可以移植，如何移植呢，我的想法是通过将应用程序打包build成对应的image（镜像），通过docker平台镜像的随处运行来实现应用程序的移植，关于理论部分就说这么多，i am still learning ,来一波实践;

​ 关于docker的安装，方法众多，因为docker是在Linux平台下面开发出来的，所以即使他提供了MAC,Linux，Windows下的相关client，Linux平台也是最佳的，下面说说如何在centos7上面安装docker:

设计模式学习笔记之工厂模式

2018-06-23

设计模式学习笔记之工厂模式

什么是工厂模式模式

创建一个接口,但由其子类决定实例化哪个类,将对象的创建延迟到其子类(来源于headFirst设计模式)

我个人解析:工厂是用来创建产品的,不同的产品需要不同的工厂,因此需要创建接口,通过接口的不同实现,来创建不同的工厂,进而创建不同的产品,但工厂只能生产某一类的产品

当然也有简单工厂模式这一模式,其与工厂模式的区别在于,只有一个具体的工厂而没有对于工厂的抽象,下文会做详述.

工厂模式类图

类图解析:

Product:产品类的父接口,作用在于,让使用具体的产品的对象依赖于该接口而非具体的产品对象

ConcreteSubject:具体的产品

Factory:所有工厂类的父类,实现所有不变的方法(在具体的工厂中,抽离出来在具体的工厂中也不会发生改变的方法)

ConcreteFactory:具体的工厂类,负责某个或者多个具体产品类的创建

对于简单工厂,其与工厂的类图的差别在于简单工厂模式没有类Factory

示例

场景

奔驰汽车需要在西安建立轮胎制造厂,现在奔驰母公司只把A系列(小型单箱车)的高端和终端两种类型的轮胎交由西安代工厂制造,用程序模拟改场景;

需求分析

简单来讲,现在由两种类型的轮胎需要西安制造厂来制造,我们可以建立一个工厂实现对这两种轮胎的制造

详细设计

工厂开工

现在需求发生了变更:总部要把C级别(小型轿车)的车的轮胎制造也要放在西安,C级别由三种高端,中端,低端,如果按照之前的方法来创建的话,我们会发现工厂里面的if-else判断会越来越多,如果总部再把其他车系的轮胎再继续放在西安,一来不符合单一职责原则,二来会造成工厂类越来越厚重,不容易维护,如何解决这个问题呢?

设计模式学习笔记之装饰器模式

2018-06-23

设计模式学习笔记之装饰器模式

什么是装饰器模式

装饰器模式是在既不改变原有的代码结构也不使用继承的情况下,动态的扩展类的功能的一种设计方式;

说白了就是,原有的功能不能再适应当前的业务,但是当前的业务和之前的还有一定的关联,稍加扩展就可以实现现有的功能,此时原有的代码逻辑不能修改,要实现该功能时使用的一种设计方法

装饰着模式通用结构图(类图)

如上图所示:

装饰器与被装饰者拥有共同的父类Component,装饰其与Component以组合的方式耦合,实现了其子类,即被装饰着与装饰器的松耦合(装饰器与抽象类耦合,并不与具体的实现类耦合)

装饰者模型示例

客户需求

有一个歌唱比赛,某选手只会唱歌(唱得很好),不会说开场白(是不是很扯淡,我觉得也是),但是他有一个孪生兄弟会读开场白,歌唱比赛必须开场白与唱歌同时具备,现在需要一种解决方案,使得唱歌比赛进行下去.

需求分析

该选手不会说开场白,但是会唱歌,他的孪生兄弟可以说开场白,这样我们可以把他的兄弟包装一下,让他上去表演,说开场白(装饰模块),轮到演唱部分的时候由选手后台假唱((⊙o⊙)…)

详细设计

实现

总结

装饰器模式遵循开放封闭的原则(扩展开放,修改封闭),实现了在不修改原实现类结构的的基础之上对类功能的扩展.

虽然给该类添加了父类,但是使得装饰者与被装饰者之间的松耦合,使得程序的可扩展性增强;

定义中的继承指代的是向下继承,是将原实现类作为父类,新的类继承该类,在子类中实现扩展,与给实现类添加抽象父类并不一样

什么是设计模式

设计模式（Design Pattern）在百度百科中的定义为:是一套被反复使用、多数人知晓的、经过分类的、代码设计经验的总结,说的通俗一点就是用来解决某些问题的套路;

设计模式遵从的原则有哪些

  1. 单一职责原则 查看详情
  2. 里氏代换原则
  3. 依赖倒置
  4. 接口隔离原则
  5. 迪米特法则(最少知道原则)
  6. 开放封闭原则(扩展开放,修改封闭)

常用的设计模式有哪些(java)

从用途的不同,可以将设计模式分成三大类:创建类模式,结构类模式,行为类模式
